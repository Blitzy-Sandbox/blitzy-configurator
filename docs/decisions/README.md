# StrikeForge — Decision Log

## Purpose

This document is the single source of truth for the non-trivial decisions that shaped the StrikeForge project across both its backlog-authoring phase and its implementation phase. The epic and story files under `/tickets/epics/` and `/tickets/stories/` deliberately carry no rationale in their bodies — they state goals, personas, capabilities, and acceptance criteria only. The implementation source files under `backend/`, `frontend/`, and the root manifests (`docker-compose.yml`, `cloudbuild.yaml`, `delivery-pipeline/clouddeploy.yaml`, etc.) likewise carry no embedded rationale per the user-provided Explainability Rule, which mandates that "rationale must NOT be embedded in code comments — the decision log is the single source of truth for 'why'." Every answer to the question "why did we choose this scope, this split, this estimate, this library, or this configuration?" lives in the table below. Backlog artifacts and implementation files cross-reference back to this log whenever a decision materially shaped their content.

## How to Read This Log

The decision log is a Markdown table with four columns: **Decision** (what was chosen, with ticket ID references whenever the choice shaped a specific epic or story), **Alternatives** (the options that were considered and rejected), **Rationale** (the reasoning that made the chosen option preferable), and **Risks** (the residual exposure carried by the choice, with mitigation where applicable). Other artifacts that need to cite a specific decision do so by quoting a distinctive leading phrase from the **Decision** cell — for example "the decision titled 'No bidirectional traceability matrix is emitted for this deliverable'" — so that citations remain stable when rows are added or reordered. A new row is added whenever a competent reviewer could reasonably have chosen differently; every such choice must be traceable here rather than buried in commit messages or ticket bodies.
